# 『kaiminちゃんのねむり丘タウン』詳細設計書

## 1. 文書目的

本書は `基本設計書.md` をもとに、MVP実装で参照する詳細設計を定義する。対象は Next.js / React / TypeScript / Vercel / Redis 構成で実装するWebブラウザ向けゲーム『kaiminちゃんのねむり丘タウン』である。

本書では、画面、API、Redisキー、データ型、主要処理、エラー、テスト観点を実装担当者が判断できる粒度まで具体化する。

## 2. 技術前提

| 項目 | 採用方針 |
| --- | --- |
| フレームワーク | Next.js App Router |
| 言語 | TypeScript |
| UI | React |
| API | Next.js Route Handlers |
| 認証 | 独自ID・パスワード認証、HttpOnly Cookie |
| パスワードハッシュ | Argon2idを優先。環境制約がある場合はbcrypt |
| データストア | Vercel連携Redis |
| Redis接続 | REST APIまたはRedis URL接続 |
| テスト | Vitest、Playwright |
| 時刻 | サーバー時刻のみ使用 |

### 2.1 Redis環境変数

Vercelに連携するRedisの環境変数は以下を使用する。

```text
KV_REST_API_READ_ONLY_TOKEN
KV_REST_API_TOKEN
KV_REST_API_URL
KV_URL
REDIS_URL
```

読み取り専用で成立する処理は `KV_REST_API_READ_ONLY_TOKEN` を使用できる。更新処理では `KV_REST_API_TOKEN`、`KV_URL`、`REDIS_URL` のいずれかを使用する。実装では使用ライブラリに合わせて接続方式を1つに統一する。

## 4. 共通型定義

### 4.1 ID・時刻

```ts
type PlayerId = string;
type BuildingInstanceId = string;
type ResidentId = string;
type ExpeditionId = string;
type UnixTimeMs = number;
```

IDはUUID v7またはULIDを推奨する。時刻はUnix time millisecondsで統一する。

### 4.2 リソース

```ts
type ResourceId = "wood" | "food" | "ore" | "dreamCotton";

type Resources = {
  wood: number;
  food: number;
  ore: number;
  dreamCotton: number;
};
```

資源量は整数で保持する。小数を保存しない。

### 4.3 プレイヤープロフィール

```ts
type PlayerProfile = {
  playerId: PlayerId;
  displayName: string;
  townName: string;
  townRank: TownRank;
  townLevel: number;

  offlineLimitSeconds: number;
  isTownPublic: boolean;
  createdAt: UnixTimeMs;
  updatedAt: UnixTimeMs;
};
```

### 4.4 建物

```ts
type BuildingStatus = "building" | "active" | "upgrading";

type BuildingInstance = {
  instanceId: BuildingInstanceId;
  type: BuildingType;
  level: number;
  x: number;

  width: number;
```

## 6. Redis詳細設計

### 6.1 キー命名規則

| 種別 | 規則 |
| --- | --- |
| 認証 | `auth:*` |
| セッション | `session:*` |
| プレイヤー | `player:{playerId}:*` |
| 世界イベント | `world:event:*` |
| ランキング | `ranking:*` |
| 冪等性 | `idempotency:{playerId}:{requestId}` |
| レート制限 | `rate:{scope}:{key}` |

### 6.2 認証キー

#### `auth:login:{normalizedLoginId}`

値は `playerId` の文字列。

```text
auth:login:kaimin1234
019abc12-3456-7890-abcd-ef1234567890
```

#### `auth:user:{playerId}`

```json
{
  "playerId": "019abc12-3456-7890-abcd-ef1234567890",
  "passwordHash": "$argon2id$...",
  "createdAt": 1783753200000,
  "passwordUpdatedAt": 1783753200000,
  "status": "active",
  "sessionVersion": 1
}
```

`sessionVersion` は全端末ログアウト、パスワード変更、復旧コード利用時にインクリメントする。

#### `auth:recovery:{recoveryCodeHash}`

値は `playerId`。復旧コード使用後は削除する。

### 6.3 セッションキー

#### `session:{sessionTokenHash}`

```json
{

  "createdAt": 1783753200000,
  "lastUsedAt": 1783753200000,
  "expiresAt": 1786345200000,
  "sessionVersion": 1
}
```

TTLは30日。APIアクセス時に残り期限が15日未満であれば、TTLとCookie期限を30日に延長する。

## 7. 認証詳細設計

### 7.1 ログインID正規化

```text
1. 前後空白を削除
2. Unicode正規化はNFKC
3. 英字は小文字化
4. 許可文字は半角英数字、ハイフン、アンダースコア
5. 長さは4から24文字
```

### 7.2 パスワード要件

| 項目 | 条件 |
| --- | --- |
| 長さ | 8から72文字 |
| 文字種 | 制限しない |
| 禁止 | 空白のみ、ログインIDと同一 |

### 7.3 セッショントークン

| 項目 | 内容 |
| --- | --- |
| 生成 | 暗号論的に安全な乱数 |
| 長さ | 32 bytes以上 |
| Cookie保存 | 生トークン |
| Redis保存 | SHA-256などでハッシュ化した値をキーに使用 |
| 有効期限 | 30日 |

### 7.4 復旧コード

形式はユーザーが読み取りやすい4ブロック構成とする。

```text
SHEEP-KAMN-82QF-7DPA
```

復旧コードは登録完了時に一度だけレスポンスへ含める。画面遷移後に再表示しない。

```text

7. セッションCookieを設定
8. 復旧コードを返却
```

## 9. 時間経過精算詳細

### 9.1 入力

| 項目 | 内容 |
| --- | --- |
| playerId | セッションから取得 |
| now | サーバー時刻 |
| profile | lastCalculatedAt、offlineLimitSeconds |
| buildings | 建物一覧 |
| resources | 現在資源 |
| residents | 住民 |
| expeditions | 探索 |

### 9.2 出力

```ts
type SettlementResult = {
  elapsedSeconds: number;
  calculatedSeconds: number;
  gainedResources: Resources;
  completedBuildingIds: BuildingInstanceId[];
  completedExpeditionIds: ExpeditionId[];
  diary: string[];
  nextProfile: PlayerProfile;
  nextResources: Resources;
  nextBuildings: BuildingInstance[];
  nextExpeditions: Expedition[];
};
```

### 9.3 擬似コード

```text
settlePlayer(playerId, now):
  load profile, resources, buildings, expeditions, residents
  elapsed = max(0, floor((now - profile.lastCalculatedAt) / 1000))
  calculated = min(elapsed, profile.offlineLimitSeconds)

  completedBuildings = buildings where status != active and completeAt <= now
  mark completedBuildings as active

  activeBuildings = buildings where status == active after completion update
  production = calculateProduction(activeBuildings, townModifiers)
  gained = floorProduction(production, calculated)
  resources = add(resources, gained)

  claimableExpeditions = expeditions where status == running and completeAt <= now
  mark claimableExpeditions as claimable

  townStats = calculateTownStats(activeBuildings, residents)
  profile.townRank = calculateTownRank(townStats)
  profile.lastCalculatedAt = now
  profile.updatedAt = now

  diary = buildDiary(gained, completedBuildings, claimableExpeditions, residents)
  save profile, resources, buildings, expeditions atomically
  return settlement result
```

### 9.4 生産補正

```text
生産量 = 基本秒間生産量 × 建物レベル補正 × 隣接補正 × 町補正 × 計算対象秒数
```

MVPの補正値は以下とする。

| 補正 | 値 |
| --- | ---: |
| レベル1 | 1.0 |
| レベル2 | 1.2 |
| レベル3 | 1.5 |
| 道路未接続 | 0.8 |
| 町ランク slowVillage | 1.05 |
| 町ランク fluffyTown | 1.10 |

## 10. マップ・建物詳細

### 10.1 マップ座標

左上を `(0, 0)`、右下を `(9, 9)` とする。座標は整数のみ。建物サイズは `width` と `height` で表現する。

### 10.2 占有判定

```text
newRect = [x, y, x + width - 1, y + height - 1]
existingRect = [ex, ey, ex + ewidth - 1, ey + eheight - 1]
重なりあり = new.left <= existing.right
          && new.right >= existing.left
          && new.top <= existing.bottom
          && new.bottom >= existing.top
```

### 10.3 道路接続

MVPでは道路を建物種別 `road` として扱う。町役場と道路で連結されている生産施設は接続済みとする。未接続の生産施設は生産効率を0.8倍にする。

### 10.4 建物完成

建物作成時点では `status: "building"` と `completeAt` を保存する。`completeAt <= serverTime` になった場合、次回精算時に `status: "active"` へ変更する。サーバー側の定期実行は不要。

## 15. 原子処理設計

### 15.1 冪等性

更新APIは `requestId` を必須にする。処理開始時に以下のキーを `SET NX EX` で作成する。

```text
idempotency:{playerId}:{requestId}
```

TTLは5分。既に存在する場合は `CONFLICT` を返すか、保存済みレスポンスを返す。MVPでは `CONFLICT` を返す。

### 15.2 資源消費

資源消費は以下を原子的に行う。

```text
1. 現在資源を取得
2. 必要量以上か確認
3. 減算
4. 関連状態を更新
```

Redis TransactionまたはLua Scriptで実装する。

### 15.3 時間経過精算

`lastCalculatedAt` を比較し、読み込み時の値と保存直前の値が一致する場合のみ更新する。競合時は再読み込みして再計算する。
