# 『kaiminちゃんのねむり丘タウン』デザイン設計書

## 1. 文書目的

本書は、既存の `kaiminFARM` アプリのゲーム画面を、文字中心の管理UIから、生活感のある町づくりゲームUIへ拡張するためのデザイン設計を定義する。

対象は `/game` を中心としたプレイ画面、町マップ、建設、住民、探索、イベント、町訪問のUIである。既存の認証、API、Redisデータ設計は維持し、フロントエンドの表示・操作体験を段階的に置き換える。

現行方針では、kaiminちゃんは画面上のキャラクタービジュアルとしては表示しない。kaiminちゃんは、留守番日記、町役場の記録、イベント説明、通知文などのテキスト上の語り手・案内役として扱う。ゲーム画面の主役は、村や町の地形、建物、施設、イベント装飾、活動マーカーである。

## 2. 参照リポジトリ調査

### 2.1 isometric-city

参照元: https://github.com/amilich/isometric-city

確認内容:

- Next.js / TypeScript / HTML5 Canvas によるWeb向け都市建設ゲーム。
- READMEでは、等角投影レンダリング、複雑な深度ソート、レイヤー管理、画像スプライトとCanvasスプライト、車両・歩行者シミュレーション、タイル配置、モバイル向けドロワー/ツールバーが主な特徴として説明されている。
- ライセンスは MIT License。

本作への適用:

- 10x10の平面グリッドを、等角投影の町マップとして表示する。
- 建物、道路、住民、装飾、エフェクトをレイヤー分離し、Y座標と高さに基づく描画順で奥行きを表現する。
- PCはサイドパネル、スマートフォンは下部ドロワーで、建設・移動・住民・探索を操作する。
- 交通や複雑な群衆AIはMVP対象外とし、住民の短い散歩、吹き出し、待機モーションに絞る。

### 2.2 farmhand

参照元: https://github.com/jeremyckahn/farmhand

確認内容:

- 農業テーマのリソース管理ゲーム。
- READMEでは、デスクトップ/モバイル対応、30秒から30分の短時間プレイ、PWA、安定性と一貫性を重視した継続開発が説明されている。
- ゲームデザイン文書では、資源・市場・意思決定を分かりやすい農場テーマで包み、短いターンで戦略判断できる設計が示されている。
- ライセンスは GPL-2.0。

本作への適用:

- 資源、建設時間、探索帰還、世界イベント進捗を、常時確認できるコンパクトなHUDにする。
- 1回3から10分のプレイに合わせ、ログイン直後に「留守番日記」「受け取り可能」「次のおすすめ行動」をまとめる。
- 経済・効率情報は隠さず、建物カードや配置プレビュー上で必要資源、効果、残り時間を即時表示する。
- コード、画像、音、文言、UI構成を直接流用しない。GPL-2.0の派生物と解釈される余地を避け、参考範囲はアイデアと設計観点に限定する。

## 4. デザインコンセプト

### 4.1 目標体験

プレイヤーがゲーム画面を開いた瞬間に、「自分がいない間も、町に小さな暮らしが続いていた」と感じられるUIにする。

中核方針:

- 町マップを画面の主役にする。
- 数値は常に読めるが、画面を支配しない。
- 建設、回収、会話、探索、納品を2タップ以内で実行できる。
- オフライン報酬は単なる結果一覧ではなく、町で起きた出来事として提示する。
- かわいさはキャラクター立ち絵ではなく、町の造形、建物の丸み、灯り、活動マーカー、余白、色、手触りで表現する。

### 4.2 ビジュアルキーワード

- ねむり丘
- 雲の上の小さな町
- やわらかい朝の光
- 羊毛、木材、草地、星くず
- 生活感のある箱庭
- 放置後に少しだけ変化している町

### 4.3 避ける表現

- 濃いファンタジーRPG風の高コントラストUI。
- 情報量の多すぎる業務ダッシュボード。
- 画面全体を同系色だけでまとめる単調な配色。
- ボタンやカードを過剰に丸くした幼児向け表現。
- 参照リポジトリの見た目をそのまま再現すること。

## 5. 描画ライブラリ選定

### 5.1 候補

| 候補 | 評価 | 採用判断 |
| --- | --- | --- |
| PixiJS v8 | 2D WebGLレンダラとして成熟。スプライト、フィルター、パーティクル、React連携、アセット管理が強い。等角投影の2D町マップに適合する。 | 推奨 |
| Phaser 3 | シーン、入力、アニメーション、物理、サウンドを含むゲームフレームワーク。ミニゲームや物理が増える場合は有力。 | 第2候補 |
| Excalibur.js | TypeScript向け2Dゲームエンジン。設計は扱いやすいが、箱庭の大量スプライトと演出面ではPixiJSのエコシステムが有利。 | 見送り |
| Three.js | 3D表現は強いが、現在の10x10タイル、2Dスプライト、モバイル軽量性には過剰。 | 見送り |
| HTML5 Canvas自前実装 | 依存は少ないが、フィルター、パーティクル、アセット管理、入力制御の実装負荷が大きい。 | 見送り |

### 5.2 推奨結論

描画ライブラリは PixiJS v8 を推奨する。

理由:

- 本作は2Dの箱庭・等角投影・スプライト表現が中心であり、3D物理や高度なゲームエンジン機能は不要。
- PixiJSはWebGLベースの高速な2D描画に向き、光、霧、キラキラ、収穫、建設完了などの視覚効果をフィルターやパーティクルで実装しやすい。
- 既存のNext.js / React画面に、PixiJSのCanvasをゲームマップ部分だけ埋め込める。
- UIフォーム、認証、ランキング、設定はReactのまま維持できる。

### 5.3 併用ライブラリ候補

導入時は、各パッケージの最新ライセンスを確認してから採用する。

| 用途 | 候補 | 用途 |
| --- | --- | --- |
| PixiJS本体 | `pixi.js` | マップ描画、スプライト、レイヤー、フィルター |
| React連携 | `@pixi/react` | ReactコンポーネントからPixiJSステージを管理 |
| カメラ操作 | `pixi-viewport` | パン、ズーム、境界制御、慣性スクロール |
| パーティクル | `@pixi/particle-emitter` または自前軽量実装 | 夢わた、星くず、建設完了、収穫演出 |
| アニメーション | PixiJS AnimatedSprite / spritesheet | 建物稼働、草木の揺れ、イベント装飾 |
| サウンド | `@pixi/sound` またはWeb Audio API | 決定音、回収音、環境音 |

## 7. PixiJSマップ設計

### 7.1 座標系

既存データ:

- `MAP_WIDTH = 10`
- `MAP_HEIGHT = 10`
- 建物は `x`, `y`, `width`, `height`
- 住民は `x`, `y`

表示座標:

```text
screenX = (x - y) * tileWidth / 2
screenY = (x + y) * tileHeight / 2
```

推奨初期値:

- `tileWidth = 96`
- `tileHeight = 48`
- PC標準ズーム: `1.0`
- Mobile標準ズーム: `0.72`
- 将来拡張時の最大マップ: 20x20まで見込む

### 7.2 レイヤー

下から順に描画する。

| レイヤー | 内容 |
| --- | --- |
| `backgroundLayer` | 空、雲、遠景、季節背景 |
| `groundLayer` | 草地タイル、道、水辺、影 |
| `previewLayer` | 建設/移動プレビュー、配置可否ハイライト |
| `buildingLayer` | 建物、施設、装飾 |
| `activityLayer` | 住民の活動マーカー、掲示板、訪問者の気配 |
| `effectLayer` | 収穫、夢わた、建設完了、イベント演出 |
| `uiOverlayLayer` | 吹き出し、選択リング、残り時間バッジ |

深度ソート:

```text
sortKey = tileY * 1000 + tileX * 10 + objectHeight
```

2x2建物は占有矩形の最奥タイルを基準にする。住民は足元座標を基準にする。

### 7.3 タイル状態

| 状態 | 表示 |
| --- | --- |
| 通常 | 草地、薄い境界 |
| 配置可能 | 淡い青緑の半透明ハイライト |
| 配置不可 | 赤系の斜線ハイライト |
| 選択中 | 白い選択リング、軽いパルス |
| 建設中 | 足場、半透明建物、残り時間バッジ |
| 強化中 | 建物上部の工具アイコン、進捗バー |
| 生産可能 | 建物上に小さな資源アイコン |

### 7.4 カメラ操作

PC:

- ドラッグでパン。
- ホイールでズーム。
- 建物選択時はインスペクタだけ更新し、カメラは自動移動しない。
- 「町役場へ戻る」ボタンで中央へ復帰。

スマートフォン:

- 1本指ドラッグでパン。
- ピンチでズーム。
- タップで選択。
- 長押しで移動候補表示。
- ボトムシート展開時もマップ上部は見える高さを残す。

## 8. アートディレクション

### 8.1 スタイル

推奨は、低解像度ドット絵ではなく、等角投影のソフト2Dスプライトとする。輪郭は細く、陰影は柔らかく、建物と地形の形状は小さな画面でも識別できるようにする。

理由:

- 町全体の柔らかい世界観と相性が良い。
- PixiJSのスプライト、フィルター、パーティクルを活かしやすい。
- 建物の成長、季節イベント、町の灯りや装飾差分を管理しやすい。

## 9. UIコンポーネント設計

### 9.1 Header HUD

表示内容:

- 町名
- 町ランク
- 木材、食料、鉱石、夢わた
- オフライン上限の残り感
- 更新
- 設定

仕様:

- 資源はアイコン+数値で表示する。
- 数値更新時は増加分を小さく浮かせる。
- スマートフォンでは資源を横スクロールせず、2列グリッドまたは折りたたみ表示にする。

### 9.2 Action Rail / Action Bar

主要アクション:

- 建設
- 移動
- 住民
- 探索
- イベント
- 訪問

PCでは左レール、スマートフォンでは下部バーに表示する。ボタンはアイコン主体にし、選択中のみラベルを表示する。

### 9.3 Inspector

選択対象ごとの表示:

| 対象 | 表示内容 | 操作 |
| --- | --- | --- |
| 空きタイル | 座標、建設候補 | 建設モードへ |
| 建物 | 名前、Lv、状態、効果、残り時間 | 強化、移動、詳細 |
| 住民 | 名前、種族、性格、なかよし度、状態 | 会話、詳細 |
| 探索隊本部 | 探索先、必要食料、帰還予定 | 探索開始、報酬受取 |
| 世界イベント | 進捗、納品資源、自分の貢献 | 納品 |

### 9.4 留守番日記モーダル

ログイン後、`offlineReport.diary` がある場合に表示する。

構成:

- 上部に町役場の記録、掲示板、手紙を連想させる見出し。
- 獲得資源のまとめ。
- 建設完了、探索帰還、住民の行動をタイムライン化。
- 「まとめて受け取る」ではなく、既にサーバー反映済みであることを前提に「町へ行く」とする。

### 9.5 建設パレット

建物カード表示:

- 建物スプライト。
- 名前。
- 必要資源。
- 建設時間。
- 生産または町評価効果。
- 配置サイズ。

配置操作:

1. 建物を選ぶ。
2. マップに半透明プレビューを表示。
3. 配置可能なら緑、不可なら赤でタイル表示。
4. タップ/クリックで確認。
5. API成功後、建設中スプライトへ切り替え。

### 9.6 住民パネル

住民はキャラクタースプライトとしては表示しない。マップ上では、住宅の灯り、畑の作業マーク、広場の吹き出し、探索中アイコンなど、町の活動マーカーとして存在を示す。

表示:

- 住宅、畑、公園などに出る短い活動マーカー。
- 広場や掲示板に出る短い吹き出し。
- なかよし度ゲージ。
- 探索中の場合は不在バッジ。

会話:

- 住民選択後、Inspectorに会話ボタン。
- 会話後はマップ上の活動マーカーや下部ログに反映する。

### 9.7 探索パネル

表示:

- 探索先の小さな背景カード。
- 必要食料。
- 所要時間。
- 報酬候補。
- 派遣可能住民。

探索中:

- 帰還予定時刻。
- 残り時間。
- 住民カードに「探索中」表示。
- 帰還可能になったらHeaderまたはAction Barに通知点を表示。

### 9.8 世界イベントパネル

表示:

- イベント名。
- 期間。
- 全体進捗バー。
- 自分の貢献量。
- 納品資源と数量入力。
- ランキング上位。

演出:

- 納品成功時、対象資源アイコンがイベント会場へ飛ぶ。
- 進捗が節目を超えたら広場の装飾が増える。

### 9.9 町訪問

訪問先の町も同じPixiJSマップコンポーネントで読み取り専用表示する。

制約:

- 建設、移動、強化操作は非表示。
- 住民会話は不可。
- 「いい夢」リアクションのみ可能。
- 自分の町と混同しないよう、Headerの色とラベルを変える。

## 10. 状態管理設計

### 10.1 既存APIとの関係

既存の `GameState` を描画用ViewModelへ変換する。

```text
GameState
  -> MapViewModel
    -> TileView[]
    -> BuildingView[]
    -> ResidentView[]
    -> EffectEvent[]
```

APIレスポンスの型は維持し、PixiJSコンポーネントはサーバー状態を直接変更しない。建設、移動、強化などは既存Route HandlerへPOSTし、成功後に返却された `GameState` を再描画する。

### 10.2 クライアント内状態

保持する状態:

- カメラ位置、ズーム。
- 選択中タイル。
- 選択中建物。
- 選択中住民。
- 操作モード: `inspect`, `build`, `move`, `visit`
- 建設プレビュー。
- 一時エフェクト。
- 開閉中パネル。

保持しない状態:

- 資源残高の正。
- 建設完了判定の正。
- 探索報酬の正。
- イベント貢献量の正。

ゲームの正は引き続きサーバー側に置く。

## 11. アニメーション・演出設計

### 11.1 常時演出

- 草と木がゆっくり揺れる。

- 町役場の旗が動く。
- 住宅や畑に小さな活動マーカーが点灯する。

### 11.2 操作演出

| --- | --- |
| 建設開始 | 足場が出現し、木材アイコンが吸い込まれる |

### 11.3 パフォーマンス方針

- 画面外の住民、エフェクト、吹き出しは更新頻度を下げる。
- スプライトはspritesheet化し、HTTPリクエスト数を抑える。
- パーティクルは同時表示数を制限する。
- `prefers-reduced-motion` が有効な場合は、揺れ、粒子、浮遊演出を減らす。
- モバイル低スペック端末では、影とフィルターを簡略化する。

## 13. アクセシビリティ

PixiJSマップはCanvas描画のため、React側に補助的なHTML情報を持つ。

対応:

- 選択中建物、住民、タイルの情報をInspectorでテキスト表示する。
- 主要操作はキーボードでも実行できる。
- 資源、建設、探索、イベント進捗はCanvas外のHUDにも表示する。
- 色だけで配置可否を伝えず、アイコンとテキストも併用する。
- 画面読み上げ用に、町の状態要約を `aria-live` で更新する。

  ### 17-X. 未決事項に対する決定事項
  1. 正式アートを自作するか、商用アセットを購入するか
推奨: 「自作ベース＋部分的にCC0/フリー素材で補完」のハイブリッド

タイル・地形：Kenney.nl（CC0、商用可、著作権表示不要）の等角投影アセットパックが非常に豊富。町マップの土台はここから流用可能。
建物・町の活動マーカー：町の造形がゲームの主役になるため、建物、広場、畑、公園、掲示板、灯り、作業マーカーは自作または世界観に合う素材を選定する。住民やkaiminちゃんのキャラクター立ち絵は作らない。建物は最初は簡易な自作シルエット（円・四角の組み合わせ）でも、色とモーションで十分「町っぽさ」は出せます。
エフェクト（星くず、夢わた等）：PixiJSのパーティクルで単色の円やグラデーションを使えば、画像アセット不要で作れる。

Kenney系を使う場合も THIRD_PARTY_NOTICES.md に必ず記載（3.4節の管理表と統合）。

2. @pixi/react vs 命令的管理
推奨: 命令的管理（生の pixi.js をuseEffect/useRefでラップ）

@pixi/react はメジャーバージョン更新でAPIが大きく変わることがあり、個人開発で追従コストを払うのはつらい。
町マップは「Reactの状態変化→Pixiの再描画」という単方向フローがはっきりしているので、useRef でPixi Applicationを持ち、useEffect でGameStateの差分をPixi側に反映する薄いブリッジ層（pixi-town-map.tsx 内に閉じ込める）で十分要件を満たせます。
Reactの外側（Inspector, HUD等）は普通のReactコンポーネントのままにできるので、責務分離もしやすい。
依存ライブラリが1つ減る＝バンドルサイズとメンテコストが減るのもメリット。

3. BGM/SEをMVPに含めるか
推奨: MVPでは含めない（Phase 3以降、しかも最小限に）

4. PWA化をUI刷新と同時に行うか
推奨: 同時にはやらない。Phase完了後の独立タスクにする

5. マップの将来最大サイズ
推奨: 20x20を上限の目安としつつ、実装は可変長で作っておく

7.1節で「将来拡張時の最大マップ: 20x20」とすでに見込みがあるので、それを正式決定値にしてよいと思います。理由：

10x10→20x20は4倍のタイル数（100→400）。PixiJSのスプライトバッチ描画＋画面外カリングを最初から実装しておけば、モバイル低スペック機でも問題になりにくい規模。
それ以上（30x30等）になると、パフォーマンスチューニング（カリング精度、テクスチャアトラス最適化）が本格的に必要になり、個人開発の工数対効果が悪化する。


実装上は「MAP_WIDTH/HEIGHTを定数ではなく設定値として持つ」程度の対応に留め、実際に20x20へ拡張するのはユーザー数やゲーム進行が進んでから判断でよいでしょう（今すぐ実装する必要はない）。
