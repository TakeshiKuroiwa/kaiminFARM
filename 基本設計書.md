# 『kaiminちゃんのねむり丘タウン』基本設計書

## 1. 文書目的

本書は、企画提案書 `GamePlan.md` をもとに、Webブラウザ向け放置型生活・町づくりシミュレーションゲーム『kaiminちゃんのねむり丘タウン』のMVP開発に必要な基本設計を定義する。

対象範囲は、アカウント登録、ログイン、セッション管理、オフライン生産、町マップ、建物、住民、探索、世界イベント、他プレイヤーの町訪問の初期実装とする。

## 2. ゲーム概要

| 項目 | 内容 |
| --- | --- |
| タイトル | kaiminちゃんのねむり丘タウン |
| ジャンル | 放置型生活・町づくりシミュレーション |
| 対応環境 | PC・スマートフォンのWebブラウザ |
| プレイ形式 | 基本無料・非同期オンライン |
| 1回の想定プレイ時間 | 3から10分 |
| メインキャラクター | 羊のキャラクター「kaiminちゃん」 |
| フロントエンド | Next.js / React / TypeScript |
| API | Next.js Route Handlers |
| ホスティング | Vercel |
| データストア | Redis |
| 認証 | ログインID・パスワード・HttpOnly Cookie |

## 3. コンセプト

プレイヤーは、雲の向こうにある小さな土地「ねむり丘」で、kaiminちゃんと共に町を復興する。木材、食料、鉱石、夢わたを集め、建物を建設し、住民を呼び戻し、探索や世界イベントを通じて町を発展させる。

本作の中核体験は、ゲームを閉じている間にも町の生活が進んでいたと感じられることである。ログイン時には資源獲得量だけでなく、kaiminちゃんの「留守番日記」として住民の行動、探索結果、来訪者などを表示する。

常時稼働するゲームサーバーは持たず、プレイヤーのAPIアクセス時に前回更新時刻と現在のサーバー時刻との差分を計算して、オフライン生産、建設完了、探索完了を精算する。

## 4. MVP範囲

### 4.1 実装対象

| 分類 | MVP内容 |

## 5. 基本ゲームサイクル

```text
ログイン
  ↓
サーバー時刻で時間経過を精算
  ↓
留守中の資源・建設・探索結果を反映
  ↓
kaiminちゃんの留守番日記を表示
  ↓
建物の建設・強化・移動
  ↓
住民との会話・依頼確認
  ↓
探索隊を派遣
  ↓
世界イベントへ資源を納品
  ↓
kaiminちゃんの就寝場所を選択
  ↓
ログアウトまたはブラウザ終了
```

## 6. システム構成

```text
ブラウザ
  |
  | HTTPS
  v
Vercel / Next.js
  |
  | Route Handlers
  v
Redis
```

Next.jsアプリケーションは、画面表示、ゲーム描画、API、認証処理を担う。Redisは、アカウント、セッション、プレイヤーデータ、資源、建物、住民、探索、世界イベント、ランキングを保持する。

### 6.1 Redis環境変数

Vercelに連携するRedisでは、以下の環境変数名を使用する。

```text
KV_REST_API_READ_ONLY_TOKEN
KV_REST_API_TOKEN
KV_REST_API_URL
KV_URL
REDIS_URL
```

実装では、読み取り専用処理と更新処理の用途を分けられる場合は `KV_REST_API_READ_ONLY_TOKEN` を参照する。ゲーム状態の更新、認証情報の保存、セッション更新、資源消費、イベント納品などの書き込み処理では、書き込み可能な接続情報を使用する。

## 9. 時間経過処理

### 9.1 基本式

```text
経過秒数 = 現在のサーバー時刻 - lastCalculatedAt
計算対象秒数 = min(経過秒数, offlineLimitSeconds)
獲得資源 = floor(秒間生産量 × 計算対象秒数 × 建物補正 × 町補正)
```

### 9.2 精算タイミング

以下の変更処理の直前に、対象プレイヤーの時間経過を精算する。

| 操作 | 精算対象 |
| --- | --- |
| ログイン | 資源、建設、探索、留守番日記 |
| `GET /api/game/state` | 資源、建設、探索 |
| 建物建設・強化・移動 | 資源、建設 |
| 探索開始・報酬受取 | 資源、探索 |
| 世界イベント納品 | 資源、イベント |

### 9.3 原子性

資源加算と `lastCalculatedAt` 更新は同一の原子的処理として扱う。Redis TransactionまたはLua Scriptを使い、同時リクエストによる二重加算を防ぐ。

## 10. 認証・セッション設計

### 10.1 登録

```text
1. ログインIDを正規化
2. ログインID重複を確認
3. パスワードをArgon2idまたはbcryptでハッシュ化
4. playerIdを生成
5. 認証情報をRedisへ保存
6. 初期ゲームデータを作成
7. 復旧コードを生成
8. 復旧コードのハッシュのみRedisへ保存
9. セッショントークンを発行
10. HttpOnly Cookieを設定
```

### 10.2 ログイン

```text
1. ログインIDを正規化
2. playerIdを検索
3. パスワードハッシュを取得
4. パスワードを照合
5. セッショントークンを生成
6. トークンのハッシュをRedisへ保存
7. 生トークンをCookieへ保存
```

ログイン失敗時のエラー文言は、IDの存在有無を判定されにくいように統一する。

```text
IDまたはパスワードが正しくありません。
```

### 10.3 Cookie設定

| 属性 | 値 |
| --- | --- |
| Name | `kaimin_session` |
| HttpOnly | true |
| Secure | true |
| SameSite | Lax |
| Path | `/` |
| Max-Age | 30日 |

Cookieにはゲームデータ、パスワード、プレイヤー名、資源、内部プレイヤーIDを保存しない。

### 10.4 復旧コード

メールアドレスを必須としないため、登録完了時に一度だけ復旧コードを表示する。Redisには復旧コード本体を保存せず、ハッシュ化した値のみ保存する。復旧コードを紛失した場合は原則復旧不可とし、登録時に明示する。

## 13. ゲームロジック設計

### 13.1 建物配置

建物配置時は以下を検証する。

| 検証項目 | 内容 |
| --- | --- |
| 座標 | マップ範囲内か |
| 占有 | 他建物と重なっていないか |
| 資源 | 必要資源を所持しているか |
| 条件 | 町ランクや前提建物を満たすか |
| 道路 | 必要な建物では道路接続があるか |

### 13.2 隣接効果

MVPでは、以下の隣接効果を優先実装する。

| 条件 | 効果 |
| --- | --- |
| 畑の隣に公園 | ここちよさ上昇 |
| 住宅の隣に公園 | ここちよさ上昇 |
| 採掘場の隣に住宅 | ここちよさ低下 |
| 伐採所または畑が道路未接続 | 生産効率低下 |

### 13.3 町評価

町評価は、人口、生産力、ここちよさ、にぎわい、安心度、自然度から算出する。MVPでは人口、生産力、ここちよさを主要評価として実装し、総合評価を「まどろみランク」として表示する。

```text
小さな集落 → のんびり村 → ふわふわ町 → 夢見る都市 → ねむり丘王国
```

### 13.4 留守番日記

時間経過精算結果から、以下の情報を日記文として生成する。

| 種類 | 例 |
| --- | --- |
| 資源 | 農園で食料が24個収穫されました |
| 建設 | 伐採所が完成しました |
| 探索 | 近くの林から木材が届きました |
| 住民 | モコが広場で休んでいました |
| イベント | 収穫祭の達成率が上がりました |

## 14. 不正対策

| リスク | 対策 |
| --- | --- |
| クライアント時刻改ざん | サーバー時刻のみ使用 |
| 資源改ざん | 資源増減はサーバー計算 |
| 探索報酬の再受取 | 原子的なstatus更新 |
| ログイン総当たり | IP単位・ログインID単位のレート制限 |
| セッション盗難 | 十分長いランダムトークン、HttpOnly、Secure |
| トークン漏洩時の被害拡大 | Redisにはトークンハッシュのみ保存 |
| 重複送信 | `idempotency:{playerId}:{requestId}` で制御 |
| 他人のデータ操作 | 操作対象playerIdはセッションからのみ確定 |

## 15. 非機能要件

| 項目 | 方針 |
| --- | --- |
| パフォーマンス | `GET /api/game/state` で画面表示に必要な状態をまとめて取得する |
| コスト | 常時実行サーバーを持たず、アクセス時精算を基本とする |
| 可用性 | Redis障害時はゲーム更新を停止し、再試行可能なエラーを返す |
| 保守性 | Redisキー命名を統一し、プレイヤー単位でデータを整理する |
| セキュリティ | パスワードはArgon2idまたはbcrypt、CookieはHttpOnly/Secure |
| アクセシビリティ | 無音でも遊べる設計、テキストで状態を確認可能にする |
| レスポンシブ | PCとスマートフォンの両方で操作可能なUIにする |

## 19. 実装上の注意

- Redisへ保存するセッショントークンと復旧コードは、必ずハッシュ化する。
- Cookieにはゲームデータを保存しない。
- すべての更新APIは、処理開始時に時間経過精算を実行する。
- 資源消費と状態更新は原子的に行う。
- 探索、建設、世界イベント納品は二重実行対策を入れる。
- ランダム報酬はサーバー側で決定する。
- クライアントは表示と操作要求に限定し、ゲーム結果の正本はRedisに置く。
- Redisキーはプレイヤー単位で整理し、画面単位でまとめて取得する。
