## 2. 企画コンセプト

『kaiminちゃんのねむり丘タウン』は、羊のキャラクター「kaiminちゃん」と一緒に、小さな集落を少しずつ発展させていく放置型の町づくりゲームである。

プレイヤーは木材、食料、鉱石などの資源を集め、住宅、生産施設、公共施設、公園などを建設する。

町に住民を呼び込み、生活環境を整えることで、小さな集落を一つの町へ成長させていく。

本作では、プレイヤーがゲームを閉じている間にも町の時間が進む。

ただし、サーバー上で常にゲーム処理を実行するのではない。

プレイヤーが次にアクセスした際に、前回の更新時刻と現在のサーバー時刻との差から、資源生産量、建設進行、探索結果などを計算する。

```text
前回の更新時刻：10:00
現在時刻：11:00
経過時間：60分

木材生産量：毎分2個
獲得木材：120個
```

この仕組みにより、常時稼働するゲームサーバーを持たず、VercelとRedisを中心としたサーバーレス構成で運営できる。

## 3. 本作が提供する体験

本作が目指すのは、単純に数値を増やす放置ゲームではない。

プレイヤーがしばらくゲームを離れ、再び町を訪れたときに、

> 自分がいない間にも、kaiminちゃんや住民たちが町で暮らしていた

と感じられる体験を提供する。

ログイン時には、資源獲得量だけではなく、kaiminちゃんによる「留守番日記」が表示される。

```text
kaiminちゃんの留守番日記

・農園でニンジンが24個収穫されました。
・モコさんが広場のベンチでお昼寝していました。
・星くず鉱山から鉱石が8個届きました。
・旅人が一人、町を訪れています。
```

放置時間そのものを、町の生活や物語として表現する。

## 4. ゲームの特徴

本作は、次の要素を中心に構成する。

1. 住民との生活感のある交流
2. 建物配置による町づくり
3. 時間経過による放置成長
4. 探索隊の派遣
5. 全プレイヤー共通の世界イベント
6. 他プレイヤーの町への訪問
7. kaiminちゃんとの関係性
8. 睡眠・休息・夢をテーマにした独自システム

町を効率だけで発展させるのではなく、住民にとって心地よい町をつくることを重要な目標とする。

住宅や生産施設を詰め込むだけでは町の評価は上がらない。

公園、花壇、ベンチ、街灯、カフェ、図書館などを配置することで、町の「ここちよさ」や「にぎわい」が上昇する。

## 5. 世界観

舞台は、雲の向こうに存在する小さな土地「ねむり丘」。

かつて、ねむり丘には多くの動物たちが暮らしていた。

しかし、ある日発生した「大あくび嵐」によって、建物や道路が吹き飛ばされ、町は小さな集落へと戻ってしまった。

町に残っていたのは、一軒の小屋と羊のkaiminちゃんだけだった。

kaiminちゃんは町長を自称しているものの、難しい仕事を始めるとすぐ眠ってしまう。

プレイヤーはkaiminちゃんの「まちづくりパートナー」となり、資源を集め、建物を修復し、住民を呼び戻していく。

町が発展すると、周辺に存在する新しい地域へ探索できるようになる。

* まどろみの森
* ふわふわ草原
* 星くず鉱山
* 月明かりの湖
* 夢わた雲海
* おやすみ雪原

それぞれの地域には、固有の資源、住民、建物、物語が存在する。

## 9. 資源システム

基本資源は以下の4種類とする。

| 資源  | 主な用途         | 主な入手方法          |
| --- | ------------ | --------------- |
| 木材  | 建物、家具、道路     | 伐採所、森の探索        |
| 食料  | 住民、料理、探索     | 畑、果樹園、釣り場       |
| 鉱石  | 建物強化、道具      | 採掘場、鉱山探索        |
| 夢わた | 特殊建築、装飾、イベント | kaiminちゃん、夢関連施設 |

資源は建物によって時間経過で生産される。

```text
獲得量
＝ 基本生産速度
× 経過時間
× 建物補正
× 町全体の補正
```

### 計算例

```text
伐採所レベル2

基本生産速度：木材2個／分
経過時間：60分
建物補正：1.2倍
町補正：1.1倍

2 × 60 × 1.2 × 1.1
＝ 木材158個
```

小数点以下は切り捨てる。

## 10. オフライン生産

無制限に資源を生産するとゲームバランスが崩れるため、オフライン生産時間には上限を設ける。

| 町の状態   | 生産上限 |
| ------ | ---: |
| 初期状態   |  8時間 |
| 倉庫レベル2 | 12時間 |
| 倉庫レベル3 | 18時間 |
| 最大強化   | 24時間 |

オフライン生産時間の上限は、倉庫や町役場の強化によって増加する。

```text
実際の生産対象時間
＝ 経過時間とオフライン上限のうち短い方
```

### 例

```text
前回アクセスからの経過時間：30時間
オフライン生産上限：12時間

生産量の計算に使う時間：12時間
```

## 12. 建物の隣接効果

建物の配置によって、生産量や町の評価が変化する。

### プラス効果

* 畑の隣に風車を置くと、食料生産量が上昇
* 住宅の隣に公園を置くと、ここちよさが上昇
* 工房の隣に倉庫を置くと、加工時間が短縮
* カフェの隣に広場を置くと、来客数が増加
* 図書館の近くに学校を置くと、町の教育評価が上昇

### マイナス効果

* 採掘場の近くに住宅を置くと、騒音によって満足度が低下
* 工房を住宅地に集中させると、自然度が低下
* 道路がない建物は生産効率が低下
* 公共施設が不足すると、人口増加が止まる

単純に強い建物を並べるだけではなく、町全体の配置を考える楽しみを作る。

町は以下の指標によって評価される。

| 指標    | 内容             |
| ----- | -------------- |
| 人口    | 町に住んでいる住民数     |
| 生産力   | 資源の総生産能力       |
| ここちよさ | 公園、自然、休憩施設の評価  |
| にぎわい  | 店舗、広場、イベントの評価  |

| 自然度   | 木、花、水辺、環境施設の評価 |

町全体の総合評価を「まどろみランク」と呼ぶ。

```text
小さな集落
  ↓
のんびり村
  ↓
ふわふわ町
  ↓
夢見る都市
  ↓
ねむり丘王国
```

ランクが上昇すると、新しい建物、住民、地域、ストーリーが解放される。

## 15. 探索システム

プレイヤーは住民を探索隊として周辺地域へ派遣できる。

### 探索の流れ

1. 探索先を選択する
2. 最大3人の住民を選ぶ
3. 食料や道具を準備する
4. 探索を開始する
5. 一定時間後に結果を受け取る

### 探索先

| ------ | ---: | -------- |
| 近くの林   |  30分 | 木材、木の実   |

## 17. 全プレイヤー共通イベント

一定期間ごとに、すべてのプレイヤーが参加できる共同イベントを開催する。

リアルタイムで同時接続する必要はない。

それぞれのプレイヤーが好きな時間に資源を納品する、非同期形式とする。

### イベント例1：大あくび雲を追い払おう

巨大な雲が太陽を隠してしまった。

全プレイヤーで夢わたを集め、雲をふかふかの枕へ変える。

```text
全体目標：夢わた10,000,000個
現在：7,420,350個
達成率：74.2%
```

### イベント例2：ねむり丘収穫祭

全プレイヤーで食料を納品する。

達成度に応じて、限定家具や衣装が解放される。

### イベント例3：星降る展望台の建設

木材と鉱石を提供し、世界共通の展望台を完成させる。

完成後は、期間限定の探索エリアが全プレイヤーに解放される。

### イベント報酬

* 参加報酬
* 個人貢献報酬
* 全体達成報酬
* 限定家具
* kaiminちゃんの衣装
* 町の記念碑
* プロフィール称号

競争よりも共同達成を重視する。

少量しか貢献できないプレイヤーでも、イベントに参加した意味を感じられる設計とする。

## 19. アカウント・認証方針

## 19.1 基本方針

本作では、GoogleアカウントやGitHubアカウントなどの外部認証を必須としない。

プレイヤーはゲーム開始時に、以下の情報を登録する。

* ログインID
* パスワード
* プレイヤー名
* 町の名前

ログインIDとパスワードを使って、別の端末やブラウザからも同じゲームデータへアクセスできる。

### 登録例

```text
ログインID：kaimin1234
パスワード：************
プレイヤー名：ふわふわ村長
町の名前：おひるね村
```

---

## 19.2 名前の使い分け

| 項目        | 用途         | 公開範囲  |
| --------- | ---------- | ----- |
| ログインID    | ログイン時の本人確認 | 非公開   |
| プレイヤー名    | ゲーム内表示     | 公開可能  |
| 町の名前      | 町の訪問画面など   | 公開可能  |
| 内部プレイヤーID | Redis上の管理  | 完全非公開 |

ログインIDとプレイヤー名は分離する。

他プレイヤーにはログインIDを表示しない。

---

## 19.3 ログイン後のセッション管理

ログインIDとパスワードは、ログイン時の本人確認に使用する。

ログイン成功後は、サーバーがランダムなセッショントークンを発行する。

```text
ID・パスワードを送信
  ↓
サーバーで本人確認
  ↓
セッショントークンを生成
  ↓
HttpOnly Cookieへ保存
  ↓
以後のAPI通信でセッションを確認
```

Cookieには、プレイヤー情報やゲームデータを保存しない。

保存するのは、ランダムなセッショントークンのみとする。

```text
kaimin_session=ランダムな文字列
```

以下の情報はCookieへ保存しない。

* パスワード
* プレイヤー名
* 所持資源
* 建物情報
* 町の進行状況
* 内部プレイヤーID

ゲームデータの正本はRedis上で管理する。

---

## 19.4 Cookieの設定

セッションCookieには、以下の設定を行う。

```text
HttpOnly
Secure
SameSite=Lax
Path=/
```

### HttpOnly

ブラウザ上のJavaScriptからCookieを読み取れないようにする。

### Secure

HTTPS通信時のみCookieを送信する。

### SameSite

外部サイトを経由した不正なリクエストを抑制する。

---

## 19.5 パスワードの保存

パスワードは平文で保存しない。

サーバー側でパスワードハッシュへ変換し、ハッシュだけをRedisへ保存する。

```text
入力されたパスワード
  ↓
パスワードハッシュ化
  ↓
Redisへハッシュだけ保存
```

保存イメージ：

```json
{
  "playerId": "019abc12-3456-7890-abcd-ef1234567890",
  "passwordHash": "$argon2id$...",
  "createdAt": "1783753200000",
  "status": "active"
}
```

パスワードハッシュには、Argon2idまたはbcryptを使用する。

SHA-256などの一般的な高速ハッシュを単独で使用しない。

---

## 19.6 アカウント復旧コード

本作ではメールアドレスを必須としない。

そのため、登録完了時に一度だけ復旧コードを発行する。

```text
アカウント復旧コード

SHEEP-KAMN-82QF-7DPA

このコードは再表示できません。
安全な場所に保存してください。
```

Redisには復旧コードそのものを保存せず、ハッシュ化した値だけを保存する。

パスワードを忘れた場合は、以下を入力して再設定する。

* ログインID
* 復旧コード
* 新しいパスワード

復旧コードを紛失した場合、原則としてアカウントを復旧できないことを登録時に明示する。

---

## 19.7 セッション有効期限

ログイン状態は、基本的に30日間維持する。

ユーザーがゲームへアクセスするたびに、一定条件で有効期限を延長する。

以下の場合は、セッションを無効化する。

* ログアウトした
* 有効期限が切れた
* パスワードを変更した
* 復旧コードを使ってパスワードを再設定した
* 運営がアカウントを停止した
* ユーザーが全端末からログアウトした

## 22. 時間経過処理

ゲーム世界を常時動作させるのではなく、APIアクセス時に時間経過を計算する。


```text
計算対象時間
＝ 経過時間とオフライン上限のうち短い方
```

```text
獲得資源
＝ 1秒あたりの生産量
× 計算対象時間
```

以下の操作時に時間経過を精算する。

* ログイン

* 探索開始

## 23. 建物の完成処理

建設中の建物には、完成予定時刻を保存する。

```json
{
  "buildingId": "farm-001",
  "type": "farm",
  "level": 1,
  "status": "building",
  "completeAt": 1783746000000
}
```

現在時刻が`completeAt`を超えていれば、次回アクセス時に完成済みへ変更する。

建設が完了した瞬間に、サーバー側で処理を実行する必要はない。

## 24. 探索の完了処理

探索開始時に以下を保存する。

```json
{
  "expeditionId": "exp-001",
  "areaId": "sleepy-forest",
  "startedAt": 1783731600000,
  "completeAt": 1783738800000,
  "status": "running",
  "rewardSeed": "random-seed"
}
```

プレイヤーが報酬を受け取る際に、現在時刻と完了時刻を比較する。

報酬の二重受取を防ぐため、受取処理は原子的に実行する。

## 30. 認証処理の流れ

## 30.1 アカウント登録

```text
1. ログインIDを正規化
2. ログインIDの重複を確認
3. パスワードをハッシュ化
4. 内部プレイヤーIDを生成
5. 認証情報をRedisへ保存
6. 初期ゲームデータを作成
7. 復旧コードを生成
8. 復旧コードのハッシュをRedisへ保存
9. セッショントークンを発行
10. セッションCookieを設定
```

## 30.2 ログイン

```text
1. ログインIDを正規化
2. Redisから内部プレイヤーIDを検索
3. パスワードハッシュを取得
4. 入力パスワードを照合
5. セッショントークンを生成
6. トークンのハッシュをRedisへ保存
7. 生のトークンをCookieへ保存
```

## 30.3 API認証

```text
1. Cookieからセッショントークンを取得
2. トークンをハッシュ化
3. Redisのセッションを検索
4. 内部プレイヤーIDを取得
5. 対象プレイヤーのゲーム処理を実行
```

クライアントから送信されたプレイヤーIDを信用しない。

操作対象のプレイヤーは、常にセッションから確定する。

## 31. 不正対策

以下の情報はクライアント側で決定しない。

* 現在時刻
* 資源獲得量
* 建設完了判定
* 探索報酬
* ランダム抽選結果
* 世界イベントへの貢献量
* ランキングスコア
* 内部プレイヤーID

### 基本対策

* サーバー時刻を使用する
* パスワードをハッシュ化する
* セッショントークンを十分長いランダム値にする
* Redisにはセッショントークンのハッシュを保存する
* ログイン回数を制限する
* 登録回数を制限する
* 資源残高をサーバー側で検証する
* 同一リクエストの再送を防ぐ
* パスワード変更時に全セッションを無効化する
* 不自然な資源増加をログへ記録する

クライアントは「何をしたいか」だけを送信する。

結果は必ずサーバー側で計算する。

## 32. ログイン試行の制限

総当たり攻撃を防ぐため、ログイン失敗回数をRedisで管理する。

### 制限例

```text
同一IPアドレス：
10分間に20回まで

同一ログインID：
10分間に10回まで
```

制限を超えた場合は、一時的にログインを拒否する。

```text
ログイン試行回数が多すぎます。
しばらくしてから、もう一度お試しください。
```

## 40. 想定される課題と対策

### パスワードを忘れるユーザーが発生する

**対策：**

* 初回登録時に復旧コードを発行する
* 復旧コードの保存を明確に案内する
* コピー機能を用意する
* 復旧コードの注意事項を表示する
* 将来的に任意のメール登録を追加する

### セッションCookieが削除される

**対策：**

* ID・パスワードで再ログインできる
* ゲームデータはRedis上に保持する
* Cookieをゲームデータの保存場所として使用しない

### ログイン攻撃を受ける

**対策：**

* IP単位のレート制限
* ログインID単位のレート制限
* エラーメッセージの統一
* 強いパスワードハッシュの利用
* セッショントークンの定期更新

### Redisの構造が複雑になる

**対策：**

* キー命名規則を統一する
* プレイヤー単位でキーを整理する
* 建物データを過度に細分化しない
* 画面単位でデータをまとめて取得する
* 古いログを一定期間で削除する

### オフライン報酬が二重加算される

**対策：**

* サーバー時刻を使用する
* 最終計算時刻の更新を原子的に行う
* 同一リクエストの再実行を防ぐ
* Redis TransactionまたはLua Scriptを利用する

### APIリクエスト数が増加する

**対策：**

* 建物ごとに個別取得しない
* 1回のゲーム状態取得でまとめて返す
* 更新処理を可能な範囲で一括化する
* 不要な自動更新を行わない
* クライアント側に短時間キャッシュを持つ

## 42. 本企画の要点

> 羊のkaiminちゃんと、小さな集落を育てる放置型町づくりゲーム。

* ログアウト中も町と資源が成長する
* 常時稼働ゲームサーバーを必要としない
* Vercel上で画面とAPIを提供する
* Redisで認証情報、ゲーム状態、イベントを管理する
* 外部アカウント認証を必須にしない
* ログインIDとパスワードでデータを引き継げる
* Cookieにはセッショントークンだけを保存する
* パスワードは安全にハッシュ化する
* メール不要の代わりに復旧コードを発行する
* 生活シミュレーションと都市育成を組み合わせる
* 睡眠、休息、夢をゲームシステムへ取り入れる
* 激しい競争ではなく、協力と箱庭づくりを中心にする
* 小規模MVPから住民、地域、イベントを追加していく

## 43. 完成イメージ

本作が目指すのは、毎日長時間ログインすることを要求するゲームではない。

プレイヤーが仕事、学校、睡眠などでゲームを離れている間にも、町ではkaiminちゃんや住民たちが生活している。

次にログインすると、資源が増えているだけでなく、町で起きた出来事が報告される。

住民が増え、建物が増え、町が少しずつにぎやかになっていく。

プレイヤーごとに建物の配置や住民との関係が異なり、それぞれ違った町の物語が生まれる。

VercelとRedisを利用することで、常時稼働する専用サーバーを持たず、小規模な状態からゲームを公開できる。

また、ID・パスワード方式を採用することで、外部サービスのアカウントを要求せず、ブラウザを変更してもゲームデータを引き継げる。

『kaiminちゃんのねむり丘タウン』は、短時間のプレイ、放置による成長、住民との交流、町の装飾、非同期共同イベントを組み合わせた、長くゆっくり育てられるWebゲームを目指す。
